### Issue #100: Code review round 13 (PR #101 — merged + deployed)

3 fixes:

1. **`is_full` never set to 1 when `maxGroupSize` reduced to equal member count** (CRITICAL) — The PATCH handler only checked `< currentMemberCount` (reject) and `> currentMemberCount && is_full` (reopen), but missed the `=== currentMemberCount` case. When an author reduced group size to exactly the current count, `is_full` stayed 0, leaving the listing visible in browse and appearing joinable. Added the missing equality branch.

2. **Transaction mutates outer `updates`/`values` arrays** (CRITICAL) — The `updates` and `values` arrays were declared outside the `db.transaction()` closure but mutated inside it via `.push()`. Structurally unsafe — any retry or double-call would corrupt the SQL with duplicate bindings. Fixed by creating local copies (`txUpdates`/`txValues`) inside the transaction.

3. **Users permanently locked out when `requires_approval` toggled off** (IMPORTANT) — When an author toggled `requires_approval` from true to false, users with pending applications became stuck: invisible to the author (gated by `requires_approval` check), blocked from direct join (pending app check), and blocked from re-apply (flag check). Fixed by auto-rejecting pending applications in the PATCH transaction when `requires_approval` is toggled off, and showing pending applicants to the author regardless of the flag.
